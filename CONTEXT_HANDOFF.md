## 1. Проект и стек

- **Путь:** `D:\Projects\Cradle\Cradle`
- **Стек:** React 19 + TypeScript + Vite + Redux Toolkit + React Router 7 + CSS Modules

### Команды / нюансы разработки
- `npm run dev` (Vite, обычно :5173; если занят — :5174).
- `npx tsc -b` — держим типы чистыми после каждого изменения.
- **⚙️ РЕЖИМ РАБОТЫ (решение владельца 2026-06-17): НЕ гонять превью-проверку после правок.** Владелец проверяет результат сам (быстрее). После изменения: только держим типы чистыми (`tsc`) и говорим «готово» — без `preview_eval`/скриншотов/навигации по превью для верификации. (Превью-инструменты использовать только если владелец прямо просит показать/проверить в браузере.)

- SPA-навигация в превью: `history.pushState(...) + dispatchEvent(new PopStateEvent('popstate'))`. React рендерит асинхронно — состояние читать **отдельным** eval-вызовом после действия.
- Console-логи превью **накапливаются и не чистятся при reload**: ошибки от промежуточных HMR-версий (по таймстемпу `?t=` в URL модуля) — мусор, можно игнорировать. Актуальность проверять по DOM / отсутствию ErrorBoundary, а не по буферу логов.

## 4. Паттерн интеграции фичи (как делаем каждую)

1. **Миграция** — таблица + RLS (select обычно `using(true)`; insert/update/delete по `auth.uid() = owner`).
2. **`lib/mapXxx.ts`** — `rowToModel` / `modelToRow` (snake_case ↔ camelCase, jsonb-поля как есть).
3. **`model/xxxThunks.ts`** — `createAsyncThunk` для load/save/create/remove; `supabase.from(...)...`; `currentUserId()` через `getSession`.
4. **`model/xxxSlice.ts`** — `extraReducers` на `.fulfilled`. Инициализация моком как fallback, реальные данные перетирают при загрузке.
5. **Загрузка** — `useEffect(dispatch(loadXxx()))` на маунте страницы, либо в `AuthBootstrap` при входе (профиль/компания/тип аккаунта).
6. **Формы редактирования** — локальный `useState(form)` + `useEffect(() => setForm(store), [store])` (синк, если данные подгрузились из БД после маунта — иначе при прямой перезагрузке страницы редактирования показывались устаревшие мок-данные).
7. **Циклы импортов** slice↔thunks (slice импортирует thunks для extraReducers; thunks импортируют actions для dispatch) — работают через live-bindings ES-модулей, т.к. используются в рантайме.

## 5. Что уже на реальном бэкенде

### Auth (`features/auth`)
- `model/authThunks.ts` — `signIn/signUp/signOut/bootstrapAuth/switchAccount` через Supabase. `signUp` кладёт `full_name` и `account_type` в `user_metadata`. Перевод ошибок на русский (`translateAuthError`).
- `lib/session.ts` — маппинг `Session` Supabase → внутренний `AuthSession`.
- `ui/AuthBootstrap.tsx` — `bootstrapAuth` + подписка `onAuthStateChange`; при входе грузит профиль (`loadProfile`) и **сохраняет аккаунт в реестр** (`accountsStore`, токены).
- `config.ts` — `AUTH_ENABLED = true`.
- `ui/AuthPage.tsx` — **редизайн: сплит-экран** (по `.claude/design_handoff_auth_redesign`). Слева форма (колонка 600px, белый фон): лого + «← На главную», табы Вход/Регистрация, заголовок+подзаголовок, **4 соц-кнопки** (VK/Google/GitHub/Yandex — пока **no-op, TODO Supabase OAuth** `signInWithOAuth`), разделитель «или по почте», сегмент **Пользователь/Компания** с иконками (только в регистрации), поля через общий `<Input/>`, **кнопка-глаз** (переключает оба поля пароля сразу), «Запомнить меня» + «Забыли пароль?» (только вход; remember пока не пробрасывается на бэк), submit, swap-ссылка, дисклеймер, подвал-ссылки. Справа (`>920px`) — `ui/GraphHero.tsx` (живой canvas-граф связей: центр + 1-й/2-й круг, дрейф узлов + бегущие сигналы, `prefers-reduced-motion` → статичный кадр) + оверлей (eyebrow, hero-текст со статами, легенда). На `≤920px` правая колонка скрыта, форма — центрированная карточка; соц-кнопки на `≤460px` в 2 ряда. **Всё поведение Redux сохранено** (`signIn`/`signUp`, валидация, redirect `<Navigate>`, `?add=1`). Валидация всплывает после blur (`touched`). `<Input/>` расширен опциональным `endAdornment` (для глаза). Старая центрированная карточка с топбаром убрана.
- Режим **`?add=1`** — форма входа при активной сессии (для «добавить аккаунт») — сохранён.
- **Смена аккаунта (как в Steam/YouTube):** `lib/accountsStore.ts` — реестр вошедших аккаунтов в localStorage (`kolibel:accounts`, refresh/access токены, имя/тип/аватар). `ui/AccountSwitcher.tsx` — модалка из меню хедера («Сменить аккаунт»): минимал — карточки аватар→имя→почта (без рамок/заливок/статусов), активный = коралловое имя, ✕ в правом верхнем углу (удалить из реестра, без переключения), коралловая ссылка «+ Добавить аккаунт» (→ `/auth?add=1`). Переключение — `switchAccount` (`supabase.auth.setSession` по сохранённым токенам → `bootstrapAuth` + `loadProfile`; устарел токен → убираем из реестра и просим войти). Токены освежаются на каждом `onAuthStateChange` (incl. `TOKEN_REFRESHED`). Старый мок-тумблер `toggleAccountType` из меню убран — тип аккаунта берётся из профиля.

- **Подход:** не жёсткие «корзины», а **единый score на пост** (cold-start версия рекомендаций; обучения на поведении пока нет — это осознанно отложено, см. ниже). Сортировка по score + **разнообразие**.
- **Сигналы score** (веса в `WEIGHTS`, легко крутить): `recency` (свежесть, ~½ за сутки), `followed` (автор в подписках), `hot` («горячесть» = (лайки + 2·комменты) с гравитацией по возрасту), `social` (**соц-доказательство** — сколько моих подписок лайкнули пост; считается из `post_likes.user_id` ∩ `followingIds`), `interest` (совпадение интересов автора с моими), `popular` (`log1p` числа подписчиков автора).
- **Разнообразие:** не больше **2** постов одного автора в «голове» ленты + интервал **3** между ними (`MAX_PER_AUTHOR`/`MIN_GAP`); избыток постов пролифик-автора уходит в **хвост по score** (ничего не теряется).
- **Откуда сигналы (всё реально, без новых таблиц):** `loadFeed` теперь кладёт в `FeedPost` → `likerIds` (из `post_likes`), `authorInterests` (навыки+должность у людей / индустрия у компаний — из `enrichAuthors`, `profiles.skills`/`companies.industry`), `authorFollowers` (один запрос-агрегат по `follows`). Мои интересы — из стора: человек = `profile.resume.skills` + должность; компания = `company.industry` + названия `directions`. `followingIds` — из `network` slice (если сеть не загружена — ранжирование мягко деградирует на свежесть+горячесть+интерес).
- **Что НЕ сделано (осознанно, cold-start):** нет обучения на поведении (что ты лайкал/у кого залипал) — на холодном старте нет данных; нет трекинга просмотров постов (популярность считается по лайкам+комментам, не по просмотрам). Заложить можно позже (таблица событий показов/кликов по постам + перевзвешивание). Веса подобраны эвристически.

#### Система рекомендаций сети — `features/network/lib/recommend.ts` (заготовка под рост базы)
- **Единый score на кандидата** (паттерн как у `feed/lib/rankFeed.ts`): взвешенная сумма сигналов; у каждого выбирается главная причина (сигнал с макс. вкладом) — используется для **разброса** (`spread` чередует причины) и упорядочивания. Веса в `PEOPLE_WEIGHTS`/`COMPANY_WEIGHTS` — легко крутить. (`RecommendationReason.kind`: `mutual|skill|role|hr|popular|vacancy|industry`.)
- **Бейдж на карточке — ТОЛЬКО «общие связи»** (по фидбеку 2026-06-16): и у людей (`person.mutual`), и у компаний (`company.fromNetwork`) показывается зелёный бейдж `N общих связей` (`mutualLabel` из `recommend.ts`, склонение) — у всех, у кого `> 0`. Остальные причины (навыки/профессия/HR/популярность/сфера/вакансии) **не отображаются** (участвуют только в скоринге/разбросе). `reasonIcons`/`ReasonIcon` из `NetworkPersonCard` удалены; у компании бейдж в `cFooter` (классы `.reason`/`.reasonMutual`).
- **Сигналы по аудитории:**
  - *Люди для пользователя:* общие навыки (`mySkills ∩ p.skills`), похожая профессия (пересечение токенов должности), HR/Founder/Lead (`p.tag`), популярность (`followerCount`, log-шкала), общие связи (`mutual`).
  - *Люди для компании:* совпадение под мои вакансии (`computeMatch` лексического движка по `myVacancies`), популярность, общие связи.
  - *Компании для пользователя:* вакансии под профессию (`computeMatch` моего резюме по вакансиям компании из `vacanciesByCompany`), общая сфера (пересечение токенов индустрии), популярность, общие связи.
  - *Компании для компании:* общая сфера/направление, популярность, общие связи.
- **Разброс** (`spread`): жадно не ставит две одинаковые причины подряд, сохраняя порядок по score → «не подряд все».
- **⚙️ Свежесть на каждую загрузку** (главная фича): (1) сид `seedRef` (`Math.random()` раз на монтаж) даёт псевдослучайный джиттер (`JITTER`), перетасовывающий близкие по score карточки — на каждую перезагрузку/возврат на страницу новый порядок; (2) **анти-повтор** `recentlyShown` (sessionStorage `kolibel:recSeen`, кольцевой буфер 24): показанные id штрафуются (`RECENT_PENALTY`) при следующей загрузке → всплывают другие. Пока пользователей мало score'ы малы → порядок фактически случайный (что и нужно); по мере роста relevance начнёт доминировать, джиттер останется для тай-брейков. **Проверено в превью:** на ремаунте порядок реально меняется.
- **Данные сигналов:** thunk `loadNetwork` добавляет `followerCount` (агрегат `follows` по followee — популярность) и `mutual`/`fromNetwork` (общие связи); причину сам **больше не проставляет** (её считает `recommend.ts`). Вакансии для матчинга грузятся в `NetworkView` (`loadVacancies`, если ещё не загружены).
- **Единый хук `features/network/lib/useRecommendations.ts`** собирает `RecContext` из стора (audience/профиль/компания/вакансии/сеть + сид свежести на монтаж + снимок `recentlyShown`) и возвращает `{ scoredPeople, scoredCompanies, recordShown }`. **Применён ВО ВСЕХ блоках рекомендаций** (по фидбеку 2026-06-16): вкладка «Сеть» (`NetworkRecommendations`), главная (`shared/ui/Recommendations/RecommendedPeople`/`RecommendedCompanies` — оба теперь `spread(scored).slice(0,N)` вместо `items.slice`), профиль (блок «Люди в твоей сфере» = тот же `RecommendedPeople`). Каждый блок при монтаже = новый сид → **новый порядок при каждом переходе/перезагрузке** (проверено: на главной порядок меняется на ремаунте; два инстанса блока на одной странице показывают разный порядок). Каждый блок один раз пишет показанные id в `recordShown` (анти-повтор).
- ⚠️ **Что осознанно НЕ сделано:** обучение на поведении, эмбеддинги (есть заготовка матчинга v2), серверный пейджинг «новых» — пока всё клиентское на загруженной выборке (`recommendedPeople` до 60, `recommendedCompanies` до 40). - **Нижний таб-бар** (`shared/ui/BottomNav`) — **Telegram-стиль** (редизайн 2026-06-17): **плавающий закруглённый блок** (`position:fixed`, `left/right:12px`, `bottom: 10px + safe-area`, `border-radius:24px`, тень, фон `--card`); 5 разделов (Главная/Вакансии/Профиль/Сеть/Чат, без поиска), у каждого **иконка сверху + название снизу**; выделенный пункт — **коралловый (`--primary`) + мягкая подложка `--primarySoft`** (скруглённая `border-radius:16px`). Рендерится из `AppHeader` → на всех авторизованных страницах. `--bottom-nav-height` поднят до **86px** на мобилке (плавающий блок 64 + отступы) — `body` `padding-bottom` и чат-высота следуют за переменной.
- **📱 App-shell на мобилке (2026-06-17, `global.css` `@media ≤980px`):** `html { height:100%; overflow:hidden }` + `body { height:100dvh; overflow-y:auto }` — **вьюпорт зафиксирован, скроллится `body`**. Зачем: иначе документ растёт по длине контента, `window.innerHeight` раздувается (на длинной главной до ~3000px), и `position:fixed` таб-бар/шапка прижимаются к раздутому низу → таб-бар «уезжает» вниз и пропадает с экрана (видно было: главная без меню, короткие вакансии — с меню). С залоченным `html` документ не растёт → фикс-элементы встают корректно на всех страницах (и в превью). **Сделано глобально через `html/body`, а НЕ через внутренний скролл `main`** — потому что профиль (`ProfileSheet`/`CompanyBrand`) и Auth не используют `<main>`, пер-страничный app-shell их бы сломал. Чат не задет (свой `height:100vh; overflow:hidden` + сам лочит `body` через JS). Клиренс под плавающий таб-бар — `body padding-bottom: var(--bottom-nav-height)+safe-area`.
- **📱 Pull-to-refresh** (`shared/ui/PullToRefresh`, смонтирован в `app/router/RootLayout`): тяга вниз от самого верха `body` (app-shell скроллит `body`) → выезжает кружок-спиннер → при перетяге порога (70px) `window.location.reload()`. Только на мобилке (`useIsMobile`), **отключён на `/chat`** (свой фикс. layout с залоченным body). Срабатывает только при `body.scrollTop===0` и если жест **не** внутри вложенного скроллера (модалки/треды — проверка `overflow-y:auto/scroll` у предков); `e.preventDefault()` глушит нативный iOS-bounce во время тяги. Состояние жеста — в `ref` (слушатели не перевешиваются на каждый кадр).
- **SupportLinks = футер**: на мобилке доступен только из бургера. Standalone-вхождения в сайдбарах обёрнуты `className="hideOnMobile"` (утилита в global.css: `display:none !important` ≤980), т.к. их класс `.footer` общий с бургером — глобально прятать нельзя.
- **Карусели рекомендаций**: `shared/ui/HScroll` (нативный CSS scroll-snap, без зависимостей). Блоки `RecommendedPeople/Companies/Vacancies` + `MySubscriptions` принимают `horizontal`, но **сами включают горизонтальный режим на мобилке** (`hcarousel = horizontal || useIsMobile()`). Слайды — `min(78%, 280px)`.
- **Вставка реко в ленты/списки**: каждые `REC_EVERY=5` элементов на мобилке — в ленте (`FeedList intersperse`, HomePage) и в списке вакансий (`VacanciesPage`). На мобилке боковые колонки этих страниц скрыты (`display:none`).
- **Чат — как в мессенджере**: на мобилке `ChatPage` одно-панельный (`useIsMobile`): список ИЛИ переписка на весь экран (`chatPanel` высотой `100dvh - header - bottom-nav - паддинги`), `ChatThread` с кнопкой «назад» (`onBack` → сброс `activeId`). **Мини-чат** (`MiniChatWidget`) на мобилке не рендерится (`if (isMobile) return null` — строго после всех хуков!).
- **Прочее**: поля форм `.row` стекуются ≤640; липкая панель «Сохранить» в редакторах поднята над таб-баром (`bottom: calc(var(--bottom-nav-height)+12px)`); дропдаун поиска на мобилке — `position:fixed` во всю ширину. Модалки уже адаптивны (`width: min(…, 100%)`).
- **⚠️ Грабли**: ранний `return` по `isMobile` ставить ТОЛЬКО после всех хуков (иначе «change in order of Hooks» при ресайзе).
